# design.md レビュー結果(2026-07-20)

`design.md` に対して 4 観点(整合性・事実確認 / 設計の妥当性・代替案 / 実装可能性 / 健全性の敵対レビュー)の
レビューを回し、上がった指摘をソースコードで検証した記録。各項目に **[検証済]**(このリポジトリのコードで
確認した)か **[未検証]**(筋は通るが未確認)を付ける。「状態」は反映状況。

## A. 実装前に必ず直す

### A1. `is_fully_unboxed` が新 `Array` を「RC 不要」と判定する [検証済] 状態: **design 反映済み**(§3.2 / §9 step 3)

**症状**: 反転後、`Array U8` / `String` / `Array (Array I64)` を含む**すべての `Array`** がリークし、
COW が発動せず、`reserve` の realloc で use-after-free になる。

**根拠**: `TypeNode::is_fully_unboxed`(`src/ast/types.rs`)は `is_box` でなければ `field_types` を再帰する。
`field_types(Array a)` は**要素型**を返す(同ファイルのコメント「For Array, return the element type」)。
よって反転後は `Array U8` -> `is_box=false`、`field_types=[U8]` -> `true`。この述語は「RC 不要」の判定として
次の 5 箇所で load-bearing:

- `src/rc_ir/rc_insert.rs::needs_rc` — `Retain`/`Release` ノードを一切出さない
- `src/object.rs::create_traverser` — `None` を返し custom traverser が生成されない
- `src/rc_ir/borrow.rs`(2 箇所)— RC unit ゼロ、ownership shape が全 `Unboxed`
- `src/rc_ir/provenance.rs::boxed_leaf_paths` — boxed leaf ゼロ

**§3.2 の「不可分 unit 述語に `is_array()` を 1 行足す」では届かない**: `borrow.rs` では
`is_fully_unboxed` の early return が `is_rc_unit_root` の判定より前にある。

**根本**: 反転後、型レベルの `field_types`(要素型 1 つ)が値レイアウト(3 field)と一致しなくなる。
現状は全 walker が `is_box` で先に止まるので露出していない。

**対応方針**: `is_fully_unboxed` を「RC unit を持たないこと」として定義し直す(最小には `Array` で常に false)。
`boxed_leaf_paths` は `is_rc_unit_root` を意図的に使っていない(union を variant へ展開するため)ので、
そちらにも独立に `Array` で止まる分岐が要る。あわせて
`debug_assert!(is_fully_unboxed(ty) == rc_units(ty).is_empty())` を置いて乖離を検知する。
「型を辿る walker は `Array` を必ず不可分 unit として止める」という不変条件を design に明文化し、
§9 step 3 のチェックリストに型駆動 walker を加える。

### A2. §13.1 の `unique_check_operand` の path/index が誤り、しかも誤ると silent に COW が消える [検証済] 状態: 未着手

**症状**: 宣言を間違えると、COW が全経路で畳まれ共有配列が in-place 破壊される。警告もパニックも出ない。

**根拠**:

- 現行 `InlineLLVMArraySetBody` は `path: vec![]`(`src/fixstd/builtin.rs`)。design は Array 系 op すべてに
  `Some{0,[0]}`(「arr の Storage leaf」)と書いており、これは誤り。`Path` の定義も「空パスは値全体」。
- `Provenance::leaf_at` は未登録 path に `unwrap_or_default()` = **空集合**を返し、`resolve_leaf` は
  空集合を **`Unique`** に解決し、`maybe_elide` は `Unique` なら無条件に `assuming_unique()` へ差し替える。
  対になる `Uniqueness::leaf_at` は `unreachable!` するのに、`Provenance` 側だけ黙って default する。
- `container_index` の規約が文書内で決まっていない。コードの index は **`free_vars_mut()` の宣言順**
  (`set` は `[array, idx, value]` なので arr = 0)だが、design は
  `_unsafe_append_value_capacity_unchecked` に「x[0]・arr[2]」と **Fix の引数順**で書いている。
- `plug` の `Some{1,[0]}` は「operand 1 = punched、その field 0 = `_arr`」で、`set` の `[0]` とは深さが
  1 段ずれている。leaf 粒度か unit 粒度かを決めていない。

**対応方針**: §13.1 の Array 系 path を `[]` に統一し、`container_index` は `free_vars_mut()` 順であることを
§13.1 の属性語彙に明記する。「不正な path は silent に `Unique` になる」を `result_prov` の `Dyn` 固定と
同格の TRAP として書く。`project_rc_unit` に「Array を降りるパスは存在しない」assert を足すと再発を防げる。

### A3. §2.2(4) の release 手順が double-free [検証済] 状態: 未着手

**症状**: refcount >= 2 の配列で共有要素が二重解放される。

**根拠**: §2.2(4) は「value の size を読み、`[0..size)` を解放して**から** field 0 の SubObject を
generic に `build_release_mark`(refcount -1、unique なら free)」と書く。refcount を見る前に要素を
無条件解放している。§3 の記述(「unique なら要素解放して free、そうでなければ -1」)と矛盾する。

**対応方針**: 正しい機構は既にある。`build_release_mark_nonnull_boxed_with(obj, work, traverse_refs)` に
**要素ループを `traverse_refs` クロージャとして渡す**(`src/object.rs` の PunchedArray arm がこの形)。
§2.2(4) を §3 に合わせて書き直し、「generic に `build_release_mark`」を削る。

## B. 仕様の誤り

### B1. `rc += n` が off-by-one、かつ「1 命令」が成立しない [検証済] 状態: 未着手

**症状**: 要素が 1 個ぶん多く retain されてリーク。`Array::fill(n, closure)` では null 参照で segfault。
`--threaded` では lost update から use-after-free。

**根拠**:

- op は `x` の所有権 1 個を受け取って n 個の参照を格納するので、正味必要なのは **`n-1`**。
  現行の正しい実装は `src/object.rs::initialize_array_buf_by_value`(ループ内 retain × n、ループ後 release × 1)。
  `n == 0`(`Array::fill(0, x)` は合法)では `x` を release する必要があり、加算では表せない。
- retain は単なるインクリメントではない: キャプチャ無しクロージャの **null 分岐**、unbox union の
  **タグ dispatch**、`refcnt_state` の **3 分岐**(LOCAL は非 atomic、THREADED は `atomicrmw`、GLOBAL は no-op)。

**対応方針**: `build_retain_n(obj, n)` を作り、既存の null 分岐 / union dispatch /
`build_branch_by_refcnt_state` を再利用して LOCAL 枝で `add n`、THREADED 枝で `atomicrmw add n`、
GLOBAL 枝で no-op にする。契約表に `n >= 0` と n=0 の扱いを書く。性能主張(n 回インクリメントしない)は保たれる。

### B2. §13.1(3) の見出しと中身が矛盾 [検証済] 状態: 未着手

見出しは「COW/uniqueness 分岐なし」だが、force-unique 分岐を持つ op が 2 つ入っている
(`_unsafe_append_value_capacity_unchecked`、`unsafe_set_bounds_unchecked`)。実装者が見出しに従うと、
COW が畳めない(性能ロス)か、COW 自体を落とす(健全性)。

### B3. `_io` 版を専用 InlineLLVM にすると borrow 窓の外へ生 ptr が漏れる [検証済] 状態: 未着手

`IO b` を返す op だと、生 ptr は返された IO アクションのクロージャに閉じ込められ、実行されるのは
op が返った後 = borrow 窓の外。既存 `borrow_boxed_io` / `mutate_boxed_ios`(`src/fixstd/std.fix`)は
内側 runner を borrow callback の**中で**走らせるか `ios` を InlineLLVM に渡し込むことで避けている。

**対応方針**: InlineLLVM は同期版と `ios` を受け取る内部版だけにし、`_io` 版は既存 2 関数と同型の
Fix-source ラッパにする。

## D. 設計判断への異議と回答

### D1. 「より安い代替(route (A) bespoke BCE / TBAA)が検討されていない」

**回答: route (A) は検討済み。** 関数型言語での本格的な BCE(プロセス間の範囲解析)は実装コストが高いと
判断し、この再設計に進んだ。加えてこの再設計は route (A) が与えないものを与える —
`push_back` の容量チェック、`reserve` の realloc、`append` の memcpy、`truncate` の O(1) 化、そして
COW/realloc を跨いで size が register に残ること。

**ただし指摘は有効**: **その判断が design.md に書かれていない**。§1 は「root cause は表現なので直し方も表現」の
1 文で代替を退けており、兄弟 doc(`dev-docs/2026-07-18-bounds-check-elim/design.md`)が
「route (A) をまず prototype すべき」と結論していることにも触れていない。§1 に判断を書く。

TBAA(struct-path alias metadata)はこのコンパイラで一切出していない [検証済] が、FAM の GEP が
struct 境界を出るケースなので効くかは疑わしい。安価な実験ではあるので記録に留める。

### D2. 認めていないコスト [未検証]

- `String` が 1 word -> 3 word。戻り値が 24 バイト集約体になる。このコンパイラは sret/byval を扱っていない
  ので、backend のメモリ経由戻り値になる可能性。`push_back` / `append` / `reserve` / `sort_by` が該当。
- union payload が 8 -> 24 バイト増。`ArrayIterator` を包む `Option` がまさにこれで、出荷済みの read-loop
  最適化の loop state を直撃する。§9 step 5 の gate に read 系(`sum_by_loop_iter` / `sum_by_loop_arr`)を
  明示すべき。
- `-O basic` / `-O none` での劣化(`_unsafe_force_unique` 撤去の代償が per-element チェックになる)。
  §5 の「ループでの check 回数は増えない」は Max 限定の主張。
- 入れ子配列のコスト記述が §2.1(+1 word、2-word 案との比較)と §9(1 -> 3 word、現状との比較)で
  基準が違う。§10 は「小さなコスト」としており過小。
- `Array a : Boxed` 削除の影響が cp-library しか調査されていない。`Document.md` / `Document-ja.md` の
  retained-ptr FFI の唯一の実例が `Array I64` で、書き換えが要る。

## F. 妥当でない / 保留と判断したもの

- **「unique = rc 1 なので 2 つの unique 値は同じ storage を指せない」が borrow 化で偽になる**(memcpy の
  正当化への異議): 結論は保たれる(mutate op に `borrows_operand` を宣言していないため)。ただし理由付けは
  弱いので、「**refcount を分岐に使う op は borrow operand を取らない**」を §5 の規則に足すのが妥当。
  同じ弱さが §7 の `borrow_elements` の理由付け(「`f` は `Ptr` しか受け取らないので array op を呼べない」)
  にもある — `f` はクロージャなので同じ配列をキャプチャできる。実際に安全なのはキャプチャ自体が retain を
  出して rc>=2 になり mutate が COW するからで、そう書き換えるべき。
- **`#ArrayStorage` を arity 0 にする案**: FAM のサイズ計算と debug 型に要素型が要るので懐疑的。要検討止まり。
- **debug info の詳細指摘** [未検証]: ただし既存 debug テスト 3 本(`print *arr`、`x/s ... + 24`、
  `<array size>` と `<array elements>` の同時表示)が落ちるのはほぼ確実。§8 は 3 行しかなく不足。
